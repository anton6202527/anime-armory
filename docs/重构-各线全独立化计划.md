# 重构：五条线全独立化计划

> 目标（用户已确认）：每条创作线（`novel` / `n2d` / `song` / `mv` / `ad`）**可单独打包分发**——
> 自带全部依赖、零外部引用。合规类（watermark AI 标识、faceswap 授权闸门）**每线各复制一份**；
> `skills/common/` 通用模块 **vendor 进每条线**。
>
> 一句话：把当前"5 条独立线 + 1 个共享底座"重构成"5 座自包含孤岛"。

---

## 0. 现状基线（已核实）

业务线之间**已经**零 Python import 耦合，`n2d/_lib/` 也已私有。真正要拆的是 3 层共享底座：

| 层 | 共享物 | 现状用法 |
|---|---|---|
| **A. shared-* skills（6）** | `shared-watermark` / `shared-video-faceswap` / `shared-image-faceswap` / `shared-refresh-candidates` / `shared-settings` / `shared-cleanup` | 各线在 compose/合规/换脸点调用（多为 CLI/人工，少量 import） |
| **B. skills/common/（~13 通用模块 + 9 个 n2d shim）** | `disclosure` `progress_md` `settings` `voice_backends` `freshness` `subtitle_render` `io_utils` `text_utils` `markdown_parser` `line_detect` `skill_snapshot` `validate_skills` … | 各线 craft/compose 脚本**真 import** |
| **C. 顶层约定文档** | `_偏好约定.md`（78 选择点目录）、`README.md`、`AGENTS.md`/`GEMINI.md`、`CLAUDE.md` | 全仓库共享真值源 |

**各线对底座的真实依赖矩阵**（vendor 时只搬自己用到的）：

| 线 | 用到的 common 模块 | 调用的 shared-* skill |
|---|---|---|
| **novel** | `disclosure` `progress_md` `freshness`(候选) `io_utils` `text_utils` | refresh-candidates、settings |
| **song** | `disclosure` `progress_md` `settings` `io_utils` | refresh-candidates、settings、(video-faceswap?) |
| **mv** | `disclosure` `progress_md` `settings` `io_utils` `subtitle_render` `freshness` | watermark、video-faceswap、refresh-candidates、settings |
| **ad** | `disclosure` `progress_md` `freshness` `subtitle_render` | watermark、video/image-faceswap、refresh-candidates、settings |
| **n2d** | `voice_backends` `subtitle_render` + 自有 `_lib/`（contract/route/profiles…）+ 9 个 common shim | watermark、video/image-faceswap、refresh-candidates、settings |

> ⚠️ 此矩阵是 ~80% 把握的勘察结果；**Phase 0 第一步就是把它逐文件钉死**（见下）。

## 1. 目标架构

每条线变成自包含目录树，约定统一为 `<line>/_lib/`（n2d 已是此形态，作模板）：

```
skills/
  n2d/_lib/            # 已存在，吸收 common shim 后自洽
  novel/_lib/          # 新建：vendor disclosure/progress/freshness/io/text + refresh/settings
  song/_lib/           # 新建
  mv/_lib/             # 新建：额外含 subtitle_render
  ad/_lib/             # 新建：额外含 subtitle_render
  n2d-watermark/       # 由 shared-watermark 复制改名（同理 mv-watermark / ad-watermark）
  n2d-faceswap/  ...   # 由 shared-*-faceswap 复制（用到的线各一份）
  <line>-craft/        # 各线 craft 内置：本线选择点目录 + refresh + settings 子命令
  README.md            # 保留（仓库索引，硬约定：改 skill 集必同步）
  # ❌ 删除：skills/common/、skills/shared-*、_偏好约定.md（拆进各线）
```

**保留的仓库级物**（非任何线的运行时依赖，属仓库工程基建，不 vendor）：
- `shared-cleanup`：纯仓库开发清理工具，只被 README 文档引用 → 降级为「仓库级 dev 工具」，不算线能力（开放项，见 §6）。
- `common/validate_skills.py` `skill_snapshot.py`：CI/维护工具 → 移到 `tools/` 或 `.claude/` 下，保留单份。
- `AGENTS.md`/`GEMINI.md`/`CLAUDE.md`：工具中立入口 + 路由表，描述「5 条独立线」，保留单份。

## 2. 分阶段执行

### Phase 0 — 冻结与清账（不改代码）
1. **钉死依赖矩阵**：对每条线 `grep -rn 'from common\|import common\|shared-'`，把 §0 矩阵补成精确到文件:行 的清单，写进 `docs/_重构清单.md`。
2. **建测试基线**：跑通当前所有 line 测试（`n2d-script` / `n2d-compose` / `n2d-image` / `novel-fetch` / 四个 review 引擎），记录绿，作为每阶段回归基准。
3. **定 vendor 约定**：统一 `<line>/_lib/` + 各脚本改用 `sys.path` 注入本线 `_lib`（参考 n2d 现有做法），避免顶层包名冲突。

### Phase 1 — 试点一条线打穿（novel，最小）
> 用最简单的线验证整套 vendor 模式 + 测试改造路径，做成模板。
1. 建 `novel/_lib/`，复制 novel 实际用到的 common 模块（disclosure/progress_md/freshness/io_utils/text_utils）。
2. 改写 novel-* 脚本里的 `from common.X` → 本线 `_lib` 导入。
3. 把 `_偏好约定.md` 里 novel 的 8 个选择点抽进 `novel-craft`（或 novel 线约定文档）。
4. novel 用到的 refresh-candidates / settings → 在 `novel-craft` 内置 `refresh`/`settings` 子命令 + 本线 `CANDIDATE_SOURCES`。
5. 跑 novel 线测试（`novel-fetch`、`novel-review`）转绿。**冻结此线为模板。**

### Phase 2 — 复制模板到 song / mv / ad
对每条线重复 Phase 1，差异点：
- **mv / ad** 额外 vendor `subtitle_render`（compose 烧字幕用）。
- **mv / ad** 各建 `<line>-watermark` + `<line>-faceswap`（从 shared-* 复制改名 + 内置 settings parser），并把 SKILL.md 里 `shared-watermark`/`shared-video-faceswap` 引用改成本线名。
- **song** 若用到 faceswap 同理；其 compose/review 与 settings 同源处改为本线 parser。
- 每条线独立跑测试转绿后再做下一条（控制爆炸半径）。

### Phase 3 — n2d 收口（最复杂，放最后）
n2d 已最私有，剩余共享是「9 个 common shim + watermark/faceswap + refresh」：
1. 把 9 个 `common/n2d_*.py` shim 删除，n2d 脚本改为直接从 `n2d/_lib/` 导入。
2. vendor `voice_backends`、`subtitle_render` 进 `n2d/_lib/`。
3. 建 `n2d-watermark` / `n2d-faceswap`（复制改名），改 `n2d-compose`/`n2d-compliance`/`n2d-voice`/`n2d-review` 引用。
4. n2d 的候选刷新内置进 `n2d-craft`（或 n2d 线对应位置）。
5. 跑 n2d 全套测试转绿。

### Phase 4 — 拆顶层文档 + 拆除死底座
1. `_偏好约定.md` 已逐线抽空 → 删除；各线选择点目录现各自归属。
2. 更新 `CLAUDE.md` 架构段（去掉「shared-* 是共享能力」「common/ 单一真值源」表述，改为「5 座自包含孤岛」）。
3. 更新 `README.md` skill 索引（**硬约定**：改 skill 集必同步）；shared-* 条目删除/改名，新增 `<line>-watermark`/`<line>-faceswap`。
4. 同步 `AGENTS.md`/`GEMINI.md` 路由表。
5. 删除 `skills/common/`（仅迁出 `validate_skills.py`/`skill_snapshot.py` 到 `tools/`）、删除 `skills/shared-*`（cleanup 见 §6）。

### Phase 5 — 独立性闸门 + 分发工具更新
1. **加独立性 CI 闸门**：一个 grep 脚本断言 `skills/<line-prefix>/` 内不出现 `from common`、不出现别线前缀、不出现 `shared-` 引用；任何回流 = 失败。这是防止以后重新长出耦合的护栏。
2. **合规平价校验（轻量保险）**：你选了「每线各复制一份」watermark/faceswap——复制会带来**合规逻辑 5 份漂移风险**。建议加一个极简 `diff` 校验脚本，断言各线 watermark 的 AI 标识核心逻辑与基准一致（不一致仅告警，不阻断）。可随时删。
3. **更新 toa / tov**：二者当前打包 `skills/ + common/ + _偏好约定.md`。改为「按线打包自包含集」——`toa` 镜像各线自治目录，`tov` 只同步 novel/n2d 两线（已自包含，无需再带 common）。
4. 跑全仓测试 + 闸门，全绿即收工。

---

## 3. 提交策略

直接进 `main`（仓库约定无 PR 流）。按阶段切提交，每条线一组：
- `refactor(novel): vendor common deps into novel/_lib, sever shared layer`
- `refactor(song|mv|ad): …`（各一）
- `refactor(n2d): inline _lib shims, vendor watermark/faceswap`
- `chore: drop skills/common + shared-* + _偏好约定, add independence gate`

每提交前跑该线测试；最后一提交跑全量 + 独立性闸门。

## 4. 风险与权衡（须知）

| 风险 | 说明 | 缓解 |
|---|---|---|
| **合规逻辑 5 份漂移** | watermark AI 标识 / faceswap 授权闸门复制 5 份，改一处易漏其他 | §5.2 平价校验脚本（告警）；改合规逻辑时全线 grep |
| **候选目录 5 份各自过期** | `CANDIDATE_SOURCES` 拆 5 份后刷新负担 ×5 | 各线 refresh 子命令 + 各自 `采集日期` 戳；`shared-refresh-candidates` 经验写进各线 reference |
| **测试导入路径全改** | 每个 pytest 按相对路径导入兄弟模块，vendor 后路径变 | 试点线先验证模式；每线改完即跑测试 |
| **重复代码维护成本** | 通用工具变 5 份 | 这是「可单独分发」的既定代价；用独立性闸门换取彻底解耦 |
| **大改一次性铺开易失控** | 工作量大 | 严格 Phase 1 试点→模板→逐线复制，单线绿了再下一条 |

## 5. 工作量粗估

- **试点 novel**：~0.5 人日（含趟通测试改造模式）。
- **song / mv / ad 各**：~0.5–1 人日/线（mv/ad 多 watermark+faceswap+subtitle）。
- **n2d**：~1 人日（shim 多、引用点多，但 _lib 已成形）。
- **文档拆除 + 闸门 + toa/tov**：~0.5–1 人日。
- **合计**：约 4–6 人日，强烈建议**逐线落地、逐线提交**，不要一把梭。

## 6. 开放项决议（已拍板 2026-06-13）

1. ✅ **`shared-cleanup` 降级为仓库级 dev 工具**，不 vendor 进任何线（迁 `tools/cleanup/` 或原位标注"非创作线能力"）。
2. ✅ **统一 `<line>/_lib/` 命名**（对齐 n2d）。
3. ✅ **`validate_skills.py` / `skill_snapshot.py` 迁 `tools/` 保留单份**。
4. ✅ **加 §5.2 合规平价校验脚本**（告警级、不阻断、可删）作为 watermark/faceswap 5 份复制的漂移保险。
