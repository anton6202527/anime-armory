# 流程自审操作手册（模式②）—— 让写歌产线可重复自我优化

把"人工复盘整条 song 线"固化成一条可复跑流程。产出 = 一份**建议报告**（默认不自动改 skill）。

## 何时跑
- 用户主动要（"song 还能优化啥""过一遍写歌流程"）。
- 做完一批歌后的阶段复盘。
- **接了新模型/能力时**（新音乐生成模型、新作词工艺、新歌声转换/换声后端、平台出了新结构/情绪控制特性）——这是最高价值触发点。

## 三轴取证（联网，必带年月）
按写歌三大验收维分轴搜，每轴落到"当前 SOTA 做法 + 证据链接 + 日期"：

| 轴 | 搜什么 | 映射到 song 的 |
|---|---|---|
| **可唱性 / 词工艺** | 平台 hook 套路、抖音/红果神曲结构、押韵/字数贴旋律、AI 作词最佳实践 | song-lyrics / songcraft.md |
| **听感 / 出歌质量** | 音乐生成 SOTA（人声真实度、风格控制、整首生成成本/速度）、prompt 组法、挑版策略 | song-compose / backends.md |
| **合规 / 可控性** | 克隆嗓 opt-in 授权动态、翻唱版权、结构/情绪可控、人声分离/换声 SOTA | song / song-cover 合规闸门 + song-cover/references |
| **模型演进**（横切） | 各 stage 当前最强模型（Suno/Udio/ACE-Step/DiffRhythm/RVC/so-vits-svc）及新语法 | 各 stage 的 backends.md / rvc.md 档案 |

> 搜索词带"2026""最新""最佳实践""翻车"更易命中实战贴；中英文各搜一轮（国内抖音/红果生态 + 英文 AI music 圈）。

## 对照 → 差距清单
逐 stage 把"基准做法"对到 `song-*/SKILL.md` + `references/*`：

- **先查已实现**：很多"新做法"产线早做了（如先写副歌 hook、可唱性自查清单、先云后本地、demucs 分离、克隆合规闸门）。**已实现的不重复立项**——只在报告里标"✅ 已覆盖"一行带过。
- **找真差距**：只记"基准有、song 没有或更弱"的。每条写成：
  ```
  差距：<一句话>
  证据：<链接>（采集 <年-月>）
  落点：<改哪个 skill 哪段 / 或新立项>
  优先级：must（影响成品质量/合规）/ optional（增稳/提效）
  可脚本化：是/否（是→能进 song_check.py）
  ```
- **分三类处置**：① 硬约束（铁律/合规）② 可选增强（opt-in 段）③ 机检项（脚本）。

## 起草 + 落地（人确认后）
1. 高价值项**起草** skill edit（写成 diff 级描述：改哪段、加什么铁律/段落）。
2. **改任何 skill → 必同步 `skills/README.md` 索引**（仓库硬约定，缺了视为未完成）。
3. **默认不自动改产线**：模式②产报告，用户拍板后再由对应 skill / 人执行编辑。**报告一次性·不留存**：只讲给用户，**不在 skill 目录存 `_流程自审_*.md`**（已 gitignore）；每次重审都重跑全流程，不依赖任何旧存档。

## 防过期 / 防噪声铁律
- 每条建议**带来源链接 + 采集日期**；旧报告里的建议可能已被采纳或已过时，落地前重新核对当前 skill。
- 容错铁律同模式①：只报"真差距"，不把"换种说法会更好"的主观偏好堆进来。
- 模型名/特性会变（如某音乐模型的整首生成上限、某换声后端的实时性）——写"能力"而非死绑某产品版本号，绑版本号的放 `backends.md`/`rvc.md` 档案里，正文写通用原则。

## 一次自审的标准产物
```
# song 流程自审 <年-月-日>
## 三轴取证摘要（含来源链接）
## 差距清单（按优先级）
  - [must] …  落点：song-xxx 某段
  - [optional] …
## 已覆盖（✅ 一行带过，证明查过没重复）
## 建议落地顺序 + 是否需要改 README
```
