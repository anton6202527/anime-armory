# song-review — 写歌质检 + 流程自审

不生产内容，只**审**。是 `song`（写歌线）家族的 QA 环节，与 `novel-review`（审小说）/ `n2d-review`（审漫剧）同构。两个模式：

- **模式①「作品质检」**——审**一首歌的产物**（`词/lyrics.md` + `歌/song.wav` + 蓝图/meta）：扫问题 → 定位（段落 / 行号 / 时间码）→ 定级 → 给修法 → 出报告。交 `mv` 做视频前 / 各阶段闸门跑。
- **模式②「流程自审」**——审**写歌流水线本身**：联网拉市场基准，对照 `song-*` 各 skill + references，产出"差距清单 + 建议改哪个 skill 哪段"。让"整套流程不断自我优化"成为一条可复跑命令。

> 写歌的三大验收维：**可唱性（词）· 听感（曲+演唱）· 合规**。本 skill 把这套体检在 song 产线里落成可跑流程。词的工艺标尺 = `song-lyrics/references/songcraft.md`；出歌后端能力 = `song-compose/references/backends.md`。

# 模式①：作品质检

## 机检 / 人判分工（照搬 n2d-review 的成熟做法）

- **机检（确定性，先跑）**：`scripts/song_check.py <写歌作品根>` —— 秒级出确定性问题：
  - **词**：段落标签解析、占位未精修（`待`/`TODO`/`（待`）、有无副歌、段落数 vs `_meta.structure` 对账、同段字数离散过大（可唱性硬信号）。
  - **音频**：`歌/song.wav` 存在性、时长 / 采样率 / 声道、**削波（clipping）**、**近静音 / 全静音**、采样率过低、目标时长窗口、头尾静音、RMS 响度代理指标（非 LUFS）。
  - **挑版 / 对账**：`歌/takes_manifest.json` 是否存在、`selected_take` 与 `song.wav` 是否能回溯、多版计划/登记/定稿快照。
  - **合规 / 对账**：`_meta.vocal_source` 合规字段缺失或标注未授权真人嗓、`rights_status` 缺失、`合规/ai_usage.json` AI 使用披露留痕、蓝图/进度文件完整性、产物快照。
  ```bash
  python3 <skill>/scripts/song_check.py <写歌作品根>          # 人读
  python3 <skill>/scripts/song_check.py <写歌作品根> --json   # 喂回 LLM 汇总
  ```
- **人判（判断题）**：机检覆盖不了的语义维度。逐维见 `references/checklist.md`。
  - **词**对照 `song-lyrics/references/songcraft.md`：押韵自然度 / 顺口不拗口 / hook 够抓耳 / 平台·曲风套路（抖音前 8 秒出 hook 等）/ 与 `创作蓝图.md` 主题情绪一致 / 原文照搬 / AI 味同质化。
  - **曲 + 演唱靠"试听"判**：LLM 听不到音频——本 skill 产出一张**试听清单**（见 checklist.md §B），由用户播放 `歌/song.wav` 逐条对照打钩：曲风贴不贴蓝图、旋律有没有记忆点 hook、人声清不清晰咬字准不准、人声与伴奏平衡、演唱情绪与词贴不贴、结构有没有对齐词。客观音频指标交机检。

## 工作流（模式①）

0. **定位 + 确认范围**：作品根 = `写歌/<曲名>/`。读 `_进度.md` 知作词/作曲/翻唱到哪步（未到的阶段不当问题报）。
1. **跑机检** → 确定性问题清单（词 + 音频 + 合规对账）。
2. **人判**：对照 `references/checklist.md` 逐维，**只记真问题**，每条带证据（词引文 + 行号 / 段落 / 音频时间码）。曲+演唱维度生成试听清单交用户。
3. **汇总报告** → 写 `写歌/<曲名>/_质检.md`：按严重度排序，每条 = 位置（`[chorus]` 第 N 行 / `song.wav` @时间码 / 文件）+ 维度 + 问题 + **修法** + 证据。附"健康度概览"表。
4. **修复回流（关键）**：歌的修法**回源头改、重跑**，不在成品 wav 上硬剪——和漫剧"回源头重跑"同理。报告里每条修法都指明**回哪个 skill 重跑**（如"词拗口→回 `song-lyrics` 改该段""削波/人声糊→回 `song-compose` 重生成或调 prompt""音色未授权→回 `song-cover`/`song-compose` 换合法音色"）。

## 严重度（定级 + 容错铁律）

| 级别 | 含 | 处置 |
|---|---|---|
| 🔴 阻断级 | 词占位未精修；演唱音色未授权真人嗓 / `vocal_source` 缺失；翻唱曲词曲版权未授权（商用）；音频全静音/损坏/不可解析；削波严重失真 | **必改**，回源头重跑 |
| 🟡 建议级 | 同段字数离散大（拗口/难唱）、押韵乱跳、hook 不抓耳、缺副歌、段落数 ≠ 蓝图结构、采样率过低、时长与目标偏离、头尾静音过长、RMS 代理响度偏低、缺 take manifest/selected_take、缺 AI 使用披露、与蓝图主题/曲风漂、人声伴奏失衡 | 建议改 |
| 🟢 润色级 | 个别词尾可更顺、桥段升华差一拍、淡出/留白偏好 | 可改可不改 |

**容错铁律**：只报"真问题"。轻微主观偏好不入报告（等同 n2d-review 容错铁律、song-lyrics 的"可唱 > 文采"）——否则噪声淹没硬伤。

# 模式②：流程自审（让写歌产线自我优化）

把"人工复盘整条 song 线"固化成可复跑流程。**节律**：用户主动要 / 做完一批歌 / 接了新作曲·作词·换声模型时跑一次。详细步骤见 `references/self_audit.md`，要点：

1. **拉基准**：联网搜当前（带年月）AI 作词 + 音乐生成 + 歌声转换主流做法，分三轴取证——**可唱性/词工艺**（hook 密度、平台套路、字数贴旋律）、**听感/出歌质量**（音乐生成 SOTA、人声真实度、风格控制、整首生成成本/速度）、**合规与可控性**（克隆授权 opt-in、结构/情绪控制、卡点/分离工具）+ 各 stage 模型演进（Suno/Udio/ACE-Step/DiffRhythm/RVC SOTA）。
2. **对照**：逐 stage 把基准 vs `song-*/SKILL.md` + `references/*` 比，找**真差距**（已做的别重复立项）。
3. **差距清单**：每条 = 差距 + 证据（带来源链接·日期）+ 落到哪个 skill 哪段 + 优先级（must/optional）+ 是否可脚本化（是→能进 `song_check.py`）。
4. **起草**：高价值项起草建议 edit；**改任何 skill 必同步 `skills/README.md` 索引**（仓库硬约定）。
5. **人确认后再写**：模式②**默认只产建议报告**，不自动改 skill。**报告是一次性的——只讲给用户、不在 skill 目录留存 `_流程自审_*.md` 这类存档**（已 gitignore）。**每次自审/重审都从头按本流程重跑**（拉基准→对照→差距），**绝不读旧报告当捷径**——市场会变，旧结论可能已过时或已落地。

> **防过期铁律**：市场建议带"采集日期 + 来源链接"，旧建议可能已被采纳或过时——写进来前先核对当前 skill 是否已有。模型名/特性会变，写"能力"而非死绑某产品版本号（绑版本号的放 `backends.md` 档案）。

## 常见错误

| 错误 | 纠正 |
|---|---|
| 只跑机检不做人判 | 机检只覆盖确定性问题；押韵/hook/曲风贴合/演唱情绪要人判（曲+演唱靠用户试听清单） |
| 只人判不跑机检 | 占位/字数离散/削波/静音/对账这类秒查，漏跑等于白审 |
| 声称"听过"音频就给曲评 | LLM 听不到 wav；曲+演唱维度必须出试听清单交用户，客观指标交机检 |
| 鸡蛋里挑骨头堆润色项 | 违容错铁律，硬伤被淹没 |
| 报问题不定位不给修法 | 必须 段落/行号/时间码定位 + 指明回哪个 skill 重跑 |
| 在成品 wav 上直接剪 | 回源头改重跑回流；成品歌是产物不是源 |
| 把未到的阶段当问题报 | 先读 `_进度.md`：还没作曲就别报"缺 song.wav" |
| 模式②直接改 skill | 默认只出建议报告，人确认后改；改 skill 必同步 README 索引 |
| 模式②抄一堆已实现的"建议" | 先核对当前 skill 是否已有；建议带来源日期防过期 |
