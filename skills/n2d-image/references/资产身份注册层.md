# 资产身份注册层（角色 ID / Face Lock / LoRA）

目标：把“定妆库”升级成**资产身份注册层**。同一角色不只是一组 PNG，还要有可被图/视频后端复用的身份记录：参考组、后端角色 ID / Face Lock / reference controls、LoRA 状态、允许角度和禁忌漂移项。

边界：本文件只管**人物角色身份**（`CHAR_xx/形态`）。场景、道具、独立服装/套装、法宝/VFX 走 `出图/共享/asset_registry.json`，详见 `资产引用注册层.md`；受伤、战损、脏污等会变化状态走 `visual_state_ledger.json`。

机器真值文件：

```
出图/共享/identity_registry.json
```

`出图/共享/prompt/00_索引.md` 继续做人读索引；`identity_registry.json` 负责 gate、批量任务和后端适配。没有 registry，就无法判断“这个镜头该喂哪组参考、哪个后端 ID、哪些角度高危、哪些漂移绝对不能放行”。

闭环报表由 `n2d-identity` 生成：

```bash
python3 skills/n2d-identity/scripts/identity.py <作品根> --write
```

输出 `生产数据/identity_adapter_matrix.json/md`（reference group / Face Lock / Character ID / LoRA 后端 binding 矩阵）和 `生产数据/identity_drift_report.json/md`（跨集角色漂移报表）。出图/出视频/审片不要只读 registry 原文，应优先读 adapter matrix；registry 是真值，matrix 是可执行视图。

---

      "forms": [
        {
          "form": "常态",

## 2. 字段规则

| 字段 | 规则 |
|---|---|
| `kind` | 必须是 `n2d_asset_identity_registry` |
| `characters[]` | 每个明确角色一条；人群剪影/无复用路人可不登记 |
| `forms[]` | 常态、觉醒态、幼年态、换装态分别登记；形态变体不能共用同一身份状态 |
| `reference_group` | 至少包含 `front/side/back/outfit/turnaround`；路径为作品根相对路径。**多形态文件名必须去重**：路径文件名必须含形态标签（如 `定妆_柳娘子_人皮态_侧.png`），不同 `form` 的同视图绝不允许同名——registry 只存路径字符串，同名会让读方无法判断"这张属于哪个形态" |
| `reference_group.expressions` | **近景高频/情绪戏角色必填**一组**同源情绪定妆**（一脸多情绪），其余角色可选。每条登记 `{emotion, path}`，emotion ∈ `中性/喜/怒/悲/惊`（按本角色戏份取子集，核心角建议 ≥3 档含中性）；**同源铁律**——必须同模型同基底、与正面定妆同一张脸，只换面部肌肉表情，脸型/五官比例/发型/痣疤不变（哭脸≠新抽一张脸）。作用：下游 `n2d-video` 近景大表情变化镜用「起表情图→止表情图」做**首尾双帧只插值**，避免模型自由生成跨情绪表情时把脸重画。若暂缺，回退裁 `定妆_<角色>_脸部特写.png`（中性），并在 `00_索引.md` 标「表情库缺，大表情近景须降级 MCU」。列表里的路径必须属于同一 `asset_key`，例如 `CHAR_03/人皮态` 只能挂 `定妆_柳娘子_人皮态_...`，不能串到其他角色/形态。命名建议 `定妆_<角色>_表情_<情绪>.png` |
| `identity_adapters.image` | 图阶段后端适配；Codex 默认无持久 ID，写 `fallback_reference_group` |
| `identity_adapters.video` | 视频阶段 Face Lock / Character ID / reference controls；无能力时写 fallback |
| `identity_adapters.lora` | LoRA 状态；`ready` 时必须写 `base_model` + `model_path` + `trigger` |
| `angle_policy` | 允许角度、高危角度、需要额外参考的动作/角度 |
| `drift_forbidden` | 审图/审片零容忍项；脸型、发型、服装主色、标志配饰、体态等 |
| `source_asset_pack` / `source_asset_slug` | 可选·角色级；`n2d-asset-market` 导入 fork 时写入：来源资产包路径 / slug（单层溯源，兼容旧导入） |
| `fork_history[]` | 可选·角色级；多级 fork 溯源链（A→B→C 不断链）。每条 `{from_pack, from_slug, from_character_id, forked_at, reason}`——键名以契约 `IDENTITY_FORK_HISTORY_ENTRY_FIELDS` 为准。导入时先继承源角色自带的链，再追加本次条目；只有 `n2d-asset-market` 写它 |

## 3. 状态枚举

推荐状态：

| 状态 | 含义 |
|---|---|
| `unsupported` | 该后端无持久身份能力 |
| `fallback_reference_group` | 使用定妆参考组 + 锚点句兜底 |
| `unregistered` | 后端支持，但本角色尚未注册 |
| `candidate` | 已决定要注册/训练，素材准备中 |
| `registered` | 已拿到后端角色 ID / handle / reference |
| `ready` | LoRA 或后端身份已可用于生产 |
| `not_needed` | 该角色不需要 LoRA 或后端 ID |

`registered/ready` 状态必须同时写实际句柄：`id`、`handle`、`reference` 或 `model_path`。否则等于假登记，gate 会阻断。

后端 mode 必须匹配能力，不能自由写：

| 区域 | 后端 | 允许 mode |
|---|---|---|
| image | `codex/openai` | `reference_group` |
| image | `kling` | `character_id` / `subject_library` / `custom_model` / `element_library` |
| image | `seedream` | `universal_reference` |
| image | `sora` | `character_cameo` |
| video | `dreamina` | `first_last_frame` / `reference_group` |
| video | `kling` | `character_id` |
| video | `seedance` | `face_lock` |
| video | `veo` | `reference_controls` |

## 4. 生成阶段怎么用

### n2d-image

1. 建共享定妆时同步创建/更新 `identity_registry.json`。
2. 跑 `n2d-identity` 生成 adapter matrix；每个含角色分镜的 `资产身份注册层` 必须写具体 `CHAR_xx/形态`，执行端从 matrix/registry 自动取 `reference_group`，而不是临场猜参考图或只靠中文角色名。
3. 多角色同框时先看 registry 是否有可用多参考/主体库 ID；没有就登记降级：分别出图 + 合成同框，或拆正反打。
4. `angle_policy.risky` 命中时，主动补侧/背/全身参考，或改分镜避开极端角度。

### n2d-video

1. 生成 `01_clips.md` 前跑 `n2d-identity`，读取 `identity_adapter_matrix.json`。
2. 若视频后端对应角色 `registered/ready`，平台参数里写入角色 ID / Face Lock / reference controls。
3. 若未注册但后端支持，先给 WARN/计划项；高危镜头（极暗、极小、极端角度、多角色接触）应优先注册。
4. 若后端无能力，明确回退到首帧 + 尾帧 + 强 `continuity` 文字。

### n2d-review

1. gate 检查 registry：字段、状态枚举、后端 mode、ready 句柄、LoRA ready 三件套。
2. 审片跑 `n2d-identity` 生成跨集漂移报表；有 `first_bad_episode` 时，修法回到该集对应镜头或共享定妆adapter 注册层。
3. 报表中的角色/集/镜头可交给 `n2d-batch` 生成定向 rerun，不整集重来。

## 5. gate 规则

`n2d-review/scripts/gate.py` 会检查：

- registry 是否存在、kind 是否正确、characters 是否非空。
- 每个角色/形态是否有 `reference_group`、`identity_adapters`、`angle_policy`、`drift_forbidden`。
- `reference_group` 的五个核心路径是否齐：front / side / back / outfit / turnaround。
- `reference_group.expressions` 是否是同角色/同形态参考，防止 `CHAR_01` 下误挂 `CHAR_03` 的脸部特写；每条是否有 `emotion`/`path`，路径是否真实存在。**warn（不硬阻断）**：被分镜标为近景高频/情绪戏的核心角缺 `expressions`（或仅中性一档）——提示「大表情近景须降级 MCU 或补表情定妆」，让 `n2d-video` 风险表据此把该角大表情镜标高危。
- 含角色分镜 prompt 是否写了具体 `CHAR_xx/形态`，否则无法自动从 registry 取定妆组。
- 正反打/反应/表情近景 prompt 是否写 `近景/反打身份锁定`，并引用脸部特写/表情参考或明确锁脸型、五官比例、发型发髻、标志配饰。
- `image/video` 每个 adapter 是否有 `mode/status`。
- `lora.status=ready` 是否写了 `base_model`、`model_path` 和 `trigger`。
- adapter `status` 是否使用枚举，`mode` 是否匹配后端能力。
- `registered/ready` 的后端身份是否写了真实 ID / handle / reference / model_path。
- 视频及之后阶段会检查核心参考图路径是否真实存在。

> 原则：后端不支持可以 fallback，但必须登记；支持却未注册不是立即阻断所有镜头，但 registry 必须让产线知道“这是未注册状态”，后续可按高危镜头/核心角色优先注册。

## 6. 和 LoRA 的关系

LoRA 仍是最重的一档，不默认启用。registry 只负责把 LoRA 的状态结构化：

- `not_needed`：短线角色或参考组已够稳。
- `candidate`：核心长线角色，参考图压不住，准备训练。
- `training`：训练中。
- `ready`：已有可用 LoRA，必须填 `model_path`、`trigger`、`base_model`，并建议带 `validation_report`、`train_job`、`model_hash`。

LoRA 具体训练执行走 `skills/n2d-lora/scripts/lora.py`；训练策略与交互判断仍读 `references/lora_consistency.md`。

## 7. 多角色同框绑定规则（崩脸最高发场景）

多数后端的单图主体锁（Face Lock / Character ID）只支持 1 个主体，多参考后端也有总图数上限（如 Seedream ≤14 图）。同框 ≥2 个注册角色时按以下**确定性规则**分配，不靠临场发挥：

1. **主体名额给最高优先角色**：按 registry 角色的 `priority` 字段（缺省时按本集出场行数多者优先，再按 registry 顺序）排序；Face Lock / Character ID / LoRA 名额只给第 1 名。镜头 prompt 必须显式标 primary（写法：`CHAR_01*`，星号=本镜主体锁持有者）。
2. **其余角色降级为参考图组**：每人最多 2 张（正面 + 当前形态服装），并在 prompt 用各自 `anchor_phrase` 锁特征词。
3. **参考图总配额裁剪**：超后端上限时，primary 角色参考组优先满额，其余按优先级均分；裁剪顺序 turnaround → 表情图 → 侧面，**正面与当前服装图永不裁**。**例外**：本镜 primary 角色且 `表情幅度=大`（要做首尾双帧的近景大表情镜）时，**该角色的起/止表情图视同正面图、不裁**——表情图正是这镜锁脸的关键料，裁了等于丢掉双帧引导。
4. **三人以上群像**：只锁前 2 优先级角色，其余靠 global_style + 锚点句兜底；该镜在 storyboard 标 `crowd: true`，QA（face_consistency / n2d-score）对 crowd 镜降低同人脸预期，避免把背景群演误报成崩脸。
5. **gate 约定**：同框镜 prompt 引用 >1 个 `CHAR_` 而没有星标 primary → warn（绑定歧义，后端会随机挑主体）。

> 设计依据：同框崩脸的根因多半是"两个角色都想锁、后端只锁得住一个、锁了谁不确定"。把"锁谁"变成确定性规则 + 把"没说锁谁"变成机检 warn，崩脸从随机事件变成可追责事件。
