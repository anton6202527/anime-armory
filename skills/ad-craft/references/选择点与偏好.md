# 拍广告 · 选择点与偏好约定（本线自包含）

> 本线（ad）私有的偏好/选择点约定。原 `skills/ad-craft/references/选择点与偏好.md` 已按线拆分，本线自带这一份，
> 不依赖任何公共层。通用机制对所有线一致；下方「选择点目录」只列本线的选择点。

# 偏好约定（所有自定义 skill 共用 · 通用机制）

> 这是**共享代码侧**的通用约定，不含任何具体用户的选择。
> 用户的实际选择是**私有的**，落在用户自己的空间，绝不写进 skill 源码。
> 任何 skill 涉及「让用户选」的地方，都按本约定读写，不要把某个平台/后端写死成唯一路径。

## 三句话原则
1. **skill 通用**：不写死平台/后端/分辨率；这些是「选择点」，由用户私有文件决定。
2. **选择私有**：用户的选择存在用户自己的空间（见下），不外发、不进共享 skill。
3. **持续记录 · 最新为准**：选择点首次遇到 → 问一次 → 写入 `_设置.md` → 同项目之后**沉默沿用**；用户随时可改（说一声或直接改文件）。**用户在任何一句输入里显式点名某选择点的值（哪怕之前已设过、哪怕在流程中途、哪怕和当前任务无关）→ 立刻以新值覆盖 `_设置.md` 对应行 + 追加一条 `## 记录`，最新选择永远优先于旧值 / 全局默认 / 缺省。** 听到即落档，不要"等下次用到该选择点再写"。

## 候选项刷新与用户输入口

选择点目录里的可选值只是**候选快照**，不是永久真值，也不是对用户输入的封死枚举。

- **可刷新来源**：候选项可来自项目 references、已有生产数据/投放回灌、模型/平台官方文档、实时搜索、以及 AI 代理的专业知识库。模型、平台、价格、规格、法规、榜单、API 能力等高变化信息，在正式花钱/不可逆/合规相关步骤前必须按需要核验；刷新时写清核验日期和来源，不能联网或不能确认时明确标注“未核验”。
- **用户输入口**：每个选择点都保留 `自定义` / `manual` / 直接手输值的入口。`skills/ad/_lib/settings.py` 的 schema 只用于安全校验和常见别名归一，不得把用户的合理新选择挡在 skill 外；确属实验值时由 agent 调用本线 `settings.py` 的 `set_project_setting(..., validate=False)` 落档，并在 `## 记录` 说明原因。本线没有独立设置 CLI。
- **菜单不等于执行逻辑**：菜单文案服务于人类决策；脚本执行时必须先把值归一成能力、参数、后端入口和降级策略。不要在任一 stage 里写 `if 菜单文案 then 唯一路径` 的硬绑定。
- **机检与刷新工具**：易变候选清单都在 `skills/ad/_lib/freshness.py:CANDIDATE_SOURCES` 登记并带 `采集日期：YYYY-MM-DD` 戳记；`python3 skills/ad/_lib/freshness.py` 机检谁过期；`本线 `_lib/refresh.py`` skill 跑「搜索核验 → 改候选 → bump 日期 + 落 provenance」的刷新流程。新增易变清单 → 登记 + 加戳，否则机检看不到。

## 适配层原则

不同选择带来的差异由适配层消化，产线正文只消费统一契约。

- **归一化**：把用户选择映射为 canonical key、能力标签（如多参考/主体库/原生音画/最大时长/分辨率上限）、实际 CLI/API/网页入口、参数包、fallback/degrade plan、合规 gate 和输出 schema。
- **专门适配**：用户选定某模型/平台/风格/规格后，下游 prompt、分镜约束、镜头时长、首尾帧/参考图策略、字幕/交付规格等都应按适配结果生成，不只是在 `_设置.md` 里记一个名字。
- **透明失败**：适配层缺能力、缺凭证、缺官方入口或候选项已过期时，停下列明缺口和可选替代；不要静默改用另一个后端，也不要把旧候选清单当作当前事实。
- **解耦边界**：候选项目录、用户私有选择、执行适配、质量/合规闸门分开维护。新增模型/平台优先补适配表或 capability profile；只有真正影响用户决策的项目才进入本文件的选择点目录。

## AI 代理交互约定 (Interactive Flow)

**原则：把枯燥的脚本命令藏在背后，让用户只做决策。**

当一个工作流包含**「机器负责结构计算（如拆分 Clip），紧接着需要大模型负责语义理解（如写 Prompt）」**的组合操作时，**绝对不要**把两段割裂的 CLI 脚本硬塞给用户让他们自己去敲。

- **正确的做法**：在第一个脚本跑完后，建立明确的**【AI 代理交互节点】**。
- **用户体验**：AI 代理应主动弹出高亮提示，或者用自然语言询问：“已完成基础切分。是否需要我启动 XXX 引擎为你自动补全提示词？”
- **后台执行**：用户回答“是”之后，AI 代理在后台默默**跑脚本 → 读取提示词 → 调用自身的 LLM 生成 JSON 结果 → 注入项目**，一气呵成。使用户感受到的是一个连贯、智能的“创作助理”，而不是一本枯燥的代码说明书。

## 两层私有存储

| 层 | 路径 | 谁写 | 作用 |
|---|---|---|---|
| **作品级** | `<作品根>/_设置.md` | skill 自动读写，用户可手改 | 本作品/曲/小说的实际选择（**权威**，覆盖全局） |
| **全局默认** | 私有全局默认 `创作偏好-默认.md`（仓库根或 `.agents/` / `.codex/` 等用户空间） | 用户维护 | 跨项目的个人默认（如「生图后端 / 视频后端 / 720p / CosyVoice / Suno」），开新项目时**预填** `_设置.md` |

- `<作品根>`：`制漫剧/<剧名>/`、`制MV/<曲名>/`、`写歌/<曲名>/`、`写小说/<项目>/` 等，与 `_进度.md` 同级。
- 全局默认使用工具中立路径：优先读仓库根 `创作偏好-默认.md`，再读 `.agents/创作偏好-默认.md`、`.codex/创作偏好-默认.md`；旧项目的 `.claude/创作偏好-默认.md` 仅作兼容读取，不作为新的共享入口。skill 文本引用「全局默认偏好文件」即指这些私有候选文件。

## 解析顺序（每次要用某选择点时）
0. **（最高优先）用户本轮输入显式指定了该选择点** → 立刻按新值覆盖 `_设置.md` + 记一行，**用新值**，跳过下面的读取。这条对**所有**选择点恒成立，且不限于"正要用到"时——只要用户一句话点名某选择，当场落档（见三句话原则 #3）。
1. 读 `<作品根>/_设置.md` 里该选择点 → 有就**直接用，不问**。
2. 没有 → 读全局默认 `创作偏好-默认.md` → 有就用其值**预填**，并**首次告知一句**（"默认 X，要改说一声"），随即写进 `_设置.md`。
3. 全局也没有 → **问用户一次**，写进 `_设置.md`。
4. 之后同项目同选择点：**沉默沿用**，不再问。
5. 无作品根的独立调用（如 novel-title / song-lyrics 单用）：能定位项目就读其 `_设置.md`，否则用全局默认 / 现问，不强行落文件。

> 例外：**合规/不可逆/花钱多**的点（真人嗓授权、版权来源、删除回流）即便记录过也要确认——安全 > 省事。
