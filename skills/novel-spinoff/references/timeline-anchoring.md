# 锚点时间线 — 并行外传的硬地基

并行外传里，"原作里该角色在 X 时刻做了 Y"是不可改的事实。新文里写到那一刻必须对齐。这一文件讲怎么把这些事实抽出来、表达成机器可消费的结构、然后在写章时使用。

## 锚点定义

**锚点 (anchor)**：原作中该配角**真正出场**的一个镜头——有动作、对话、视角带过、或被主角清楚地看见做了某事。

**提及 (mention)**：原作中只是被别人嘴里提到、被信件/卷轴写到、被旁人评价——配角本人不在场。

两者都进锚点表，但作用不同：
- anchor 是**硬约束**，新文中对应时刻必须呈现这件事。
- mention 是**软约束**，告诉续写者"别人是这么看他的"，影响角色塑造和留白填补，但不强制新文中要呈现 mention 这一刻本身。

## 锚点表格式（`设定/锚点表.json`）

```json
{
  "character": "<配角名>",
  "source_novel": "<原作路径>",
  "extracted_at": "YYYY-MM-DD",
  "anchors": [
    {
      "id": "A01",
      "type": "anchor",
      "source_chapter": 3,
      "source_excerpt_hint": "第3章约 30% 位置",
      "event": "一句话事件骨架，例：在客栈拦下主角索要丹方",
      "state": "受伤未愈，独行",
      "dialogue_points": ["索要丹方", "提到师门旧事"],
      "present_with": ["主角", "店小二"],
      "known_to_self": ["主角已得太虚剑", "自己师门被灭"],
      "unknown_to_self": ["主角真实身份是X"],
      "assigned_chapter": null
    },
    {
      "id": "M01",
      "type": "mention",
      "source_chapter": 5,
      "source_excerpt_hint": "第5章约 80% 位置",
      "event": "主角的师父向主角提及此人",
      "what_was_said": "评价此人'看似清冷实则护短'",
      "assigned_chapter": null
    }
  ]
}
```

字段说明：
- `event` —— **事件骨架**，不是原作原文。例："拦下主角索要丹方"是骨架，"她拦住林岚的去路，冷声道：'丹方留下。'"是原文片段。前者可以重新叙述，后者不能照搬。
- `state` —— 这一刻配角自己的状态（伤情/位置/同行者/已知情报）。
- `known_to_self` / `unknown_to_self` —— **极重要**。配角在这一刻**已知什么、还不知道什么**，决定续写时他的内心戏怎么写、能猜到什么、不能猜到什么。续写最易翻车的点是给配角灌入了"作者视角的已知"。
- `present_with` —— 在场其他人，决定能说什么不能说什么。
- `assigned_chapter` —— 第 3 步章纲编织时填，标这个锚点要在新文第几章被对齐。

## 抽取流程

### 粗筛（脚本，`extract_anchors.py`）

- 在原作 txt 上用正则搜配角名 + 常见称谓（"她"/"他"指代不抓，只抓显式名字）。
- 抽出每个命中点的所在章节、前后 ~300 字段落。
- 全部写进锚点表的 `candidates` 字段，**type 暂不定**，留 Claude 精筛。

### 精筛（主对话 Claude）

逐个 candidate 判断：

1. **是 anchor 还是 mention？**
   - 段落里这个角色有自己的动作/对话/明确视角 → anchor
   - 只是被别人提到、信里写到、回忆里出现 → mention
   - 极个别情况：原作用代词"他"长篇描写但根据上下文确认是该角色 → 也算 anchor，但 `event` 字段标"代词出场"

2. **填字段**：基于段落上下文填 `event` / `state` / `dialogue_points` / `present_with` / `known_to_self` / `unknown_to_self`。**不要把原作原文复制到 `event` 里**——必须重述为事件骨架。

3. **去重**：同一事件如果原作不同章节多次描写（回忆/旁人提及），合并成一条 anchor + 多条 mention（mention 指向同一 event_id）。

4. **排序**：按 `source_chapter` 升序，给定 `id`（anchor 用 A01、A02…，mention 用 M01、M02…）。

精筛完写回 `锚点表.json`，删 `candidates` 字段，只留 `anchors`。

## 锚点和章纲的绑定

第 3 步章纲编织时，把每个 anchor 的 `assigned_chapter` 填上——即"这个锚点要在新文第几章被对齐"。

绑定规则：
- 原作里两个锚点同章 → 新文可以拆成相邻两章（赴约 + 现场）。
- 原作里一个锚点跨多个原作章节（持续场景）→ 新文用一章完整覆盖。
- 早期锚点稀疏 → 新文前几章可以无锚点，纯建配角独立视角。
- 末期锚点密集 → 新文末几章靠章纲拉节奏，不要每章都对锚点。

## 续写时怎么用锚点

写第 N 章时，子代理 prompt 里要明确：
- 本章涉及的锚点 ID 列表
- 每个锚点的 event / state / known/unknown / present_with
- **对齐方式**：用自己的笔重新叙述这个事件，从配角视角，可以加内心戏 / 环境细节 / 配角的私人动机；但事件骨架不能变、对话**要点**要在（具体台词可以用配角自己的口吻重写）、在场人物名单不能多也不能少。

## 反 OOC 自检清单

每章写完，主对话用这个清单扫一遍涉及锚点的部分：

- [ ] 配角在这一刻的**已知信息**是否被违反？（没让他知道作者后面才揭示的事）
- [ ] 在场其他人是否齐全 / 多余？
- [ ] 对话**要点**是否覆盖？（不要求字字对应）
- [ ] 配角状态（伤情/位置/同行者）是否一致？
- [ ] 是否大段复制了原作原文？

任何一项不过 → 在该章 Edit 改。

## 法律边界（重申）

锚点机制是一种"工艺约束"，不是"复制原作文本"的许可证。新文中：
- 事件骨架可以一致（这是历史事实级别的对齐，不是文本复制）。
- 具体描写、对话遣词、心理铺陈都必须由你重新写。
- 引用原作原文超过几十字的，应当作为引文注明（同人小说一般不这么干，所以等价于**不要这么干**）。
