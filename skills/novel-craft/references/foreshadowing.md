# 伏笔工艺守则（埋—养—收闭环）

钩子（`hooks.md`）是短线悬念，伏笔是**跨章/跨卷的长线承诺**。读者爽感的一大来源是"原来早有铺垫"。本篇管伏笔的设计与回收追踪；落库用 `设定/foreshadowing_ledger.json`（产物 owner = `novel-wiki`，schema 见 `contract.md` 的 `foreshadowing_ledger` 边界）。`novel-review`/`novel-wiki` 按本篇查"伏笔逾期/烂尾"。

## 伏笔三态：埋（seed）→ 养（reinforce）→ 收（payoff）

- **埋**：首次不动声色地给出信息，当时看像闲笔。要"藏在功能性场景里"，别让读者察觉是伏笔。
- **养**（可选）：中途以不同形式复现一两次，加深印象，但不揭晓。
- **收**：在关键节点兑现，最好同时**成为一次反转或高潮的支点**——伏笔回收的最高级是"既解释了过去、又推动了当下"。

## ledger 每条伏笔登记的字段（落 `foreshadowing_ledger.json`）

| 字段 | 含义 |
|---|---|
| `id` | 伏笔编号 |
| `seed_chapter` | 埋设章 |
| `type` | 类型：身份/物品/能力/关系/世界观/预言 |
| `summary` | 一句话：埋的是什么 |
| `payoff_chapter` | 计划回收章（可先估） |
| `payoff_is_twist` | 回收时是否兼作反转/高潮支点 |
| `status` | seeded / reinforced / paid / **overdue** / dropped |

## 伏笔反模式（review/wiki 必扫）

- ❌ **逾期未收（overdue）**：埋了 N 章仍未回收且无养护——读者忘了，等于没埋。设阈值：超过约定跨度未 `paid` 即告警。
- ❌ **烂尾/敷衍收**：回收时草草一句带过，浪费了铺垫。
- ❌ **无埋强收**：高潮处突然"其实我早有安排"，但前文查无 seed——读者觉得作者开挂。回收必须在 ledger 里能追到 seed。
- ❌ **伏笔过载**：埋了一堆永不回收的"假伏笔"，信息噪音。埋必登记，登记必有计划回收章。
- ❌ **太明显的埋**：埋的时候用力过猛（"他不经意地看了那把剑一眼，仿佛那将决定一切"）——读者一眼识破，惊喜没了。

## 与一致性的接口

- 伏笔回收**不得与已确立设定冲突**：收的时候要回扫 `setting-bible.md`/wiki，确认揭晓的真相和此前所有描写自洽（`novel-wiki` 的硬冲突检查）。
- 角色生死/在场状态也走 wiki：别让一个已死角色来回收伏笔。
