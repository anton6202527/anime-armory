## Q1：直接把分镜剧本丢给即梦，还是先出图再生成视频？<a id="q1"></a>

**强烈建议先出图再生成视频（图生视频流程），而不是直接文生视频。**

### 1.1 人物一致性问题
直接文生视频，每个镜头的角色长相、服装、发型都会漂移，跨镜头根本对不上。先做"定妆图"锁死角色形象，再用图生视频，整集人物才稳得住。这也是本仓库 `n2d` skill 默认的工作流（角色定妆 → 分镜图 → 视频）。

### 1.2 试错成本
- **文生视频**：一条 5s 视频 几毛~几块，废一条就是几块没了，要等几十秒
- **文生图**：几分钱一张，秒出，不满意立刻重抽

把"构图、人物、光线、镜头"这些容易翻车的变量在图阶段就锁死，视频阶段只控制"动作/运镜"，废片率断崖式下降。

### 1.3 可控性
图阶段可以局部重绘、改服装、统一色调；直接出视频这些都没法精修，只能整条重抽。

### 1.4 即梦的实际表现
即梦的"文生视频"在长剧情、多角色场景下一致性较弱；它的"图生视频（首帧/首尾帧）"才是稳定出片的主力模式。专业短剧团队基本都是这个流程。

### 1.5 唯一例外
纯空镜、转场、氛围镜头（风/雨/火/光效）可以直接文生视频，省一步。

### 1.6 标准流程

```
小说原文
  → 分镜剧本（episode_N.md）
  → 角色定妆 prompt（锁死人物形象，先出定妆图）
  → 分镜图 prompt（基于定妆图，逐镜出图）
  → 视频 prompt（图生视频，控制动作/运镜）
  → 配音 + BGM + 字幕
  → 合成成片
```

## Q2：制作一集的整体步骤是什么？<a id="q2"></a>

**4 个大步骤，前后严格依赖，不能跳。**

```
Step 1 定妆（角色+场景+道具）
   ↓ 所有定妆图存到 第N集/出图/定妆_xxx.png
Step 2 分镜出图（用定妆图作参考图，逐镜出图）
   ↓ 镜头图存到 第N集/出图/镜头N_xxx.png
Step 3 图生视频（每张定稿图喂即梦"图生视频"+ 运镜 prompt）
   ↓ 视频片段存到 第N集/视频/镜头N.mp4
Step 4 合成（配音 + BGM + 字幕 + 剪辑）
   ↓ 成片
```

**关键原则：**
- 上一步不锁死，下一步全废
- Step 1 占整体时间约 30%，但决定整集质量
- 一次只做一个角色/场景的定妆，全部稳了再进 Step 2

## Q3：定妆阶段要先出哪些图？顺序是什么？<a id="q3"></a>

按 **"主角 → 配角 → 反派 → 场景 → 道具"** 顺序，原因：

1. **主角先**：主角是全集出现最多的，定妆稳了才能验证 prompt 模板是否有效
2. **主角的"形态变体"算独立定妆**：比如沈念有常态、觉醒态、银牌态，每个都要单独定妆
3. **反派最后**：反派要跟主角"截然不同"，先做完主角的脸，反派 prompt 才能针对性堵死"长得像主角"的翻车
4. **场景独立做**：场景必须用"空镜"出图，不能带任何人物（即梦经常偷偷画进一个剪影）
5. **道具最后**：因为道具是"特写"，可有可无，分镜阶段也能临时补

**第1集案例：**
```
1.1 沈念·常态定妆   ← 整集人物锚点
1.2 沈念·觉醒态定妆  ← 用 1.1 做参考图
1.3 小禾定妆        ← 不用参考图（新角色）
1.4 柳娘子定妆      ← 不用参考图（新角色，且要跟主角拉开差异）
1.5 冷宫寝殿场景图  ← 严禁带人物
1.6 道具图（赐死三件套等）← 可选
```

## Q5：定妆图怎么挑？检查清单是什么？<a id="q5"></a>

### 5.1 通用 6 条（每张定妆都要核对）

1. ✅ 是**符合设定的脸型**（古风东方脸 vs 欧美脸）
2. ✅ 发型/发色对（披发 vs 双环髻 vs 紧实发髻）
3. ✅ 服装色对（月白 vs 青色 vs 深蓝）
4. ✅ 配饰对（沈念无配饰 / 小禾无金饰 / 柳娘子有银簪）
5. ✅ 设定细节（沈念左手腕疤、小禾雀斑等，能体现最好，丢了也接受）
6. ✅ 背景是干净纯色

### 5.2 挑选优先级（4 选 1 的逻辑）

| 优先级 | 标准 | 为什么 |
|---|---|---|
| ① | **姿态最中性** | 定妆要做参考图复用，侧身/夸张姿势会让后续镜头都歪 |
| ② | **双臂/手可见** | 后面觉醒态、动作戏要画手臂图腾、抓握动作，藏手就废了 |
| ③ | **脸最像设定** | 4 张里挑最匹配人设关键词的那张 |
| ④ | 视觉冲击力 | 排最后，定妆要的不是"最美"，是"最稳" |

### 5.3 已知瑕疵可接受

定妆**不需要完美**，只需要"脸/服装/发型"三大件锚定即可。以下瑕疵接受：
- 设定中的"左手腕一道淡旧疤" → 被画成"一片红斑"（即梦理解偏差，可接受）
- "冷宫三年憔悴感"没画出来（即梦倾向于画健康脸）

**为什么不重抽？**
1. 重抽不一定更好，可能脸变了反而更亏
2. 这些小瑕疵会被后续构图/服装遮盖
3. **已经稳的脸不要轻易换**

## Q8：新角色（小禾、柳娘子）要不要用前面角色做参考图？<a id="q8"></a>

**不要。完全相反——做新角色前必须先移除前一步的参考图。**

### 8.1 后果

如果带着沈念的参考图画小禾，会出现：
- "长得像沈念的小宫女"
- 凤眼出现在小禾脸上
- 月白色衣服变成青色但脸是沈念

### 8.2 操作

每次切换角色前，**显式检查并清空上一角色参考图**，只保留当前角色的正面/侧面/背面三视图、服装参考和本镜场景参考。多参考后端都可能把上一组参考“粘”到下一张，不主动清理会串脸。

### 8.3 但同一角色的"形态变体"必须用参考图

| 场景 | 用参考图？ |
|---|---|
| 沈念常态 → 沈念觉醒态 | ✅ 用（同一个人） |
| 沈念 → 小禾 | ❌ 不用（不同人） |
| 小禾 → 柳娘子 | ❌ 不用（不同人） |
| 柳娘子伪善态 → 柳娘子死亡态 | ✅ 用（同一角色） |

## Q9：角色年龄/外貌偏离设定（比如小禾画得太小），是重抽还是接受？<a id="q9"></a>

**优先接受，除非彻底破坏戏剧逻辑。**

### 9.1 真实案例：小禾设定 15 岁，被画成 ~11 岁

第一反应是重抽。但仔细分析后接受了，理由：

1. **戏剧效果其实更好**：小禾在镜头 5/8/12 三场戏（哭喊报信/被太监捂嘴/瞪眼震惊）。一个看起来更小的忠仆，能让"太监捂嘴拖走"的暴力感**更冲击**，沈念的"护主"动机也更强。漫剧惯用手法。

2. **重抽风险高**：即梦对"15岁少女"的稳定输出就在这个面相区间。继续重抽不一定更老，但脸型可能变（鼻子嘴巴比例一变就跟之前不一致）。**已经稳的脸不要轻易换。**

### 9.2 决策框架

| 偏差类型 | 怎么办 |
|---|---|
| 服装色错了（月白画成紫色） | **重抽**，颜色错全集穿帮 |
| 发型错了（双环髻画成披发） | **重抽**，发型是辨识度核心 |
| 年龄偏小/偏大 ±3-5 岁 | **接受**，戏剧上往往更好 |
| 缺少疤痕、雀斑等小细节 | **接受**，分镜阶段补 |
| 表情风格不对（伪善画成凶相） | **重抽**，气质决定角色 |
| 多了/少了配饰 | 看是否破坏身份，破坏则重抽 |

### 9.3 如果非要再大/再小，prompt 加这句

```
14-15岁少女初长成，已有少女轮廓但保留稚气，身高比成年女子矮一头
```
负向词补 `8岁、9岁、10岁、儿童、幼儿、童星感`。

## Q10：反派（柳娘子）怎么避免被画得像主角？<a id="q10"></a>

### 10.1 负向词主动"驱赶"主角特征

柳娘子的负向词必须包含**沈念和小禾的视觉关键词**，防止即梦"风格污染"：

```
年轻少女、稚气、圆脸、双环髻、披发、月白色、淡青色、华丽贵气、
嫔妃感、皇后感、邪恶咧嘴、明显凶相、僵尸妆、妖怪外露、獠牙、
低幼Q版、卡通脸、画风漂移、多余文字水印、多余手指、现代物件、
五官崩坏、欧美脸、写真摄影、暴露服装
```

**重点：**
- "稚气、圆脸、双环髻" → 防止画成小禾
- "月白、淡青、华丽" → 防止画成沈念
- "邪恶咧嘴、獠牙" → 柳娘子是**伪善**，不能脸上写"我是妖"

### 10.2 正向词强调"区分特征"

| 主角特征 | 反派对应特征 |
|---|---|
| 20岁出头、清瘦 | **30岁中年、端庄保养** |
| 凤眼 | **鹅蛋脸、眼尾微吊** |
| 月白粗布 | **深蓝端庄绸料、盘扣宽袖** |
| 无配饰 | **素银簪、管事腰牌** |
| 披发素布带 | **乌黑一丝不苟发髻** |

每个对立面都明确说出来，即梦就不会混。

## Q12：场景图要避免哪些坑？<a id="q12"></a>

### 12.1 必须严防"画进人"

即梦做场景图时经常偷偷画进一个剪影/远景人物，破坏空镜效果。

**强制负向词：**
```
人物、人脸、女子、男子、宫女、太监、贵妃、剪影、人影
```

### 12.2 必须强调"破败 + 死寂"

即梦的默认审美偏"明亮华丽"，不主动堵会画成正常寝宫。

**强制负向词：**
```
明亮、华丽、整洁、新房、富丽堂皇、彩色、暖色调、阳光、白天、户外
```

### 12.3 必须包含"复用元素"

场景里会反复入镜的物件必须在定妆场景图里出现：
- 雕花木榻（主要走戏区）
- 角落铜镜（镜头3要用）
- 案台 + 残烛（氛围空镜要用）
- 蛛网、霉斑（破败感锚点）

定妆阶段不画进去，后续镜头就没有视觉锚定。

### 12.4 绝不带任何角色的参考图

场景定妆**必须清空参考图**。如果带着人物参考图，会出现"角色躺在床上"等莫名其妙的画面。

## Q16：换 AI 工具（即梦↔可灵↔DALL-E 等）时怎么权衡画风一致性？<a id="q16"></a>

### 16.1 各家画风指纹（实测对比）

| 工具 | 风格指纹 | 适用 |
|---|---|---|
| **即梦 图片3.0** | 国风写实、影楼级光感、皮肤通透、东方审美 | 国风短剧主力 |
| **可灵 Kling** | 写实偏冷艳、电影质感、华人脸型自然 | 国风兜底备选，最接近即梦 |
| **DALL-E 3 / gpt-image-1** | 插画感、西方审美、皮肤偏暖、面部比例西化 | 海外投放素材 |
| **Flux Pro** | 好莱坞写实、欧美审美主导、亚洲脸不够自然 | 通用配套素材 |
| **SDXL/SD3** | 高度可控但需 LoRA，原生偏二次元 | 进阶玩家 |
| **Midjourney** | 艺术感最强、电影海报感、不可控性最高 | 单图艺术品而非系列 |

### 16.2 切换工具的"一致性税"

切换工具 = 整集所有人物/场景**必须重做定妆**。原因：

- AI 模型有"美学惯性"——同一 prompt 在不同模型下出来是不同的脸/光感
- 即使保留"参考图"功能，跨模型迁移效果都很差（参考图编码方式不一样）
- 视觉一致性是漫剧的核心质感，跳变 = 观感破产

### 16.3 切换决策表

| 情境 | 推荐 |
|---|---|
| 同一集制作中 | ❌ 绝对不切 |
| 集间过渡（第N集 → 第N+1集） | ⚠️ 可切，但要重做所有定妆 |
| 整剧未开始 | ✅ 选定后不再改 |
| 主工具账号封禁/积分耗尽 | ⚠️ 紧急情况切换备选（提前备好可灵） |

### 16.4 备选策略：双工具栈

国风短剧推荐配置：
- **主**：`生图AI` 所选官方图后端（默认 Codex）+ 即梦/Seedance/可灵/Veo 图生视频
- **备**：视频后端可切可灵 Kling / Veo / Seedance；图片后端也可切官方多参考后端，但必须整集/整项目统一切，不做单镜兜底
- 切视频后端时只改锚定句和视频 prompt；若切出图后端，先统一设置并重做受影响定妆/分镜，否则一致性税最大

禁止用即梦/Dreamina 当图片后端；一致性吃紧时补定妆参考、角色 ID 资料或 LoRA，不切到即梦出图。

## Q18：图 AI、生视频模型和生视频渠道是什么关系？image prompt 该按谁写？<a id="q18"></a>

### 18.1 三项架构

```
图 AI（出图工具）→ 图片 → 生视频模型（运动估计/风格基线） → 生视频渠道（调用入口）
       ↑                          ↑                         ↑
   决定 prompt 写法        决定图片要能被谁消化        决定 CLI/API/网页
```

- **图 AI** = 出图工具选择；当前正式产线默认 Codex，可选 Dreamina/即梦官方 CLI 和官方多参考后端；禁止第三方逆向、`同视频AI` / `同视频模型` 含糊口径和 web 自动化
- **生视频模型** = 风格基线与运动估计分布（决定图片必须长成什么样子，因为它的 image2video 只熟悉自家训练分布）
- **生视频渠道** = 调用入口（例如即梦/Dreamina、豆包、海螺AI、可灵/Kling、Google Gemini API、Runway API）

### 18.2 为什么 image prompt 按生视频模型来定风格？

生视频模型的 image2video 本质：
1. 把输入图当首帧
2. 在自家训练分布里推断运动 + 续帧

如果首帧不在生视频模型训练分布里（比如 Seedance/Kling/Veo 拿到完全不匹配的图像风格），运动推断会崩——人脸抖、转场怪、丝绸飘动不自然。

所以 image prompt 描述的"图片该长啥样"必须是**生视频模型熟悉的样子**，而不是图 AI 自己的默认审美。

### 18.3 同 AI vs 跨 AI 决策表

| 组合 | 锚定句 | 备注 |
|---|---|---|
| 所选官方图后端 + Seedance 2.0 via 即梦/Dreamina | ✅ **必须拼** Seedance/即梦分布的图像风格锚定句 | 默认正式链路 |
| 所选官方图后端 + Veo 3.1 via Gemini API | ✅ 必须拼 Veo 的英文锚定句 | 海外短剧 |
| Dreamina/即梦官方 CLI + Seedance 2.0 via 即梦/Dreamina | ✅ 可用 | 需显式 `生图AI: Dreamina/即梦`，同集不混后端 |

### 18.4 锚定句拼接位置

```
image prompt = 主体 + 外貌锚定 + 动作 + 环境 + 景别 + 画风词 + [生视频模型锚定句]
                                                          ↑
                                                       接在画风词之后
```

锚定句**不要**插进主体描述里（会扰乱构图），就放最后做"风格说明"。

### 18.5 同 AI 是最佳选择吗？

**是。** 理由：
- 同厂/同分布训练数据闭环，无翻译损耗
- 跨家用锚定句桥接是 90% 还原，不是 100%
- 同厂通常还有"参考图机制"互通（即梦的参考图机制喂自家图最稳）

跨家方案存在的价值是**省钱**（比如 Gemini 订阅免费出图，只在视频阶段付费）。

### 18.6 在 skill 里怎么落地？

- `global_style.md` 顶部记三项（"目标视频模型" + "生视频渠道" + "目标图AI"）
- 角色卡的 ① 定妆照 prompt 默认按生视频模型风格写
- 跨 AI 时所有 image prompt 末尾追加生视频模型的锚定句（见 `references/platforms.md`）
- 已建好的角色卡末尾要有"跨 AI 用法"区块（含中英双版锚定句），方便手动拼接

## Q19：定妆图能跨集复用吗？目录怎么组织？<a id="q19"></a>

### 19.1 必须复用，否则全集穿帮

沈念在 100 集里是同一张脸、同一套服装；冷宫寝殿在 5 集里是同一处场景。这些是**全篇共享资产**，绝不按集切分。

### 19.2 目录架构（v3，2026-05-31 起，4-skill 拆分后）

共享层 vs 本集层 分离；每个 skill 自占一个顶层文件夹：

```
作品根/
├── 设定库/                    ← 跨阶段设定资产
│   ├── characters/            ← 角色卡（设定）shared，由 n2d-script 写
│   └── locations/             ← 场景卡 shared，由 n2d-script 写
└── 出图/                      ← Stage 4: n2d-image 产物
    ├── 共享/                  全篇共享定妆库（旧项目 common/ 仅读取兼容）
    │   ├── prompt/
    │   │   ├── 00_索引.md
    │   │   ├── 角色定妆.md
    │   │   ├── 场景定妆.md
    │   │   └── 道具定妆.md
    │   └── 图片/
    │       └── 定妆_*.png     定妆 PNG 库
    └── 第N集/
        ├── prompt/             本集 prompt 文件
        │   ├── 00_总览.md      本集图清单（引用 shared）
        │   └── 01_分镜出图.md  本集独有分镜 prompt
        └── 图片/
            └── 镜头N_*.png     本集分镜 PNG
```

> 历史记录：v2（2026-05 起）共享层在 `分镜剧本/出图/` + `分镜剧本/出图prompt/`，本集在 `分镜剧本/第N集/出图prompt/` 等。v3 起每个 skill 占一个顶层文件夹，**全篇共享资产从 stage-1 子目录里迁出独立**，详见 references/architecture.md 目录铁律。

### 19.3 决策表：什么放共享？什么放每集？

| 资产 | 放哪 | 理由 |
|---|---|---|
| 角色定妆 PNG/prompt | 共享 | 角色全集统一 |
| 角色形态变体（觉醒态/银牌态）| 共享 | 多形态，跨集复用 |
| 场景定妆 | 共享 | 场景通常多集复用 |
| 反复入镜道具 / HUD 系统光幕 | 共享 | 全集统一视觉 |
| 一次性道具（仅本集出现）| 本集分镜 | 不复用，无需独立块 |
| 分镜出图 | 本集 | 一镜一图，不跨集 |
| 封面 | 本集 | 一集一封 |
| 死亡形态 / 仅本集形态（如柳娘子第1集就死）| **仍放共享** | 一致性 > 节省空间，规则统一才好维护 |

### 19.4 工作流（新角色/场景/道具首现时）

```
1. 第N集 出料发现新角色/场景/道具
2. 出图/共享/prompt/00_索引.md 追加 ⬜ 行
3. 出图/共享/prompt/角色|场景|道具定妆.md 追加完整 prompt 块
4. 跑 `生图AI` 所选官方/已登录后端生图 → 挑图 → PNG 落 出图/共享/图片/；Dreamina/即梦官方 CLI 可用，第三方逆向/web 自动化禁用
5. 索引状态改 ✅
6. 出图/第N集/prompt/00_总览.md 引用共享层
7. 后续集自动复用，不再重做
```

### 19.5 进度算法（`_进度.md` 的 `出图` 列）

```
出图 列 = 已存在 PNG / 本集需要的所有 PNG（含共享复用 + 本集分镜）
```

举例 — 第2集（假设需要：沈念常态 ✅ + 沈念觉醒态 ✅ + 小禾 ✅ + 冷宫寝殿 ✅（第1集做的）+ 偏殿修炼室 ⬜ + 8 张分镜）：
- 总数 = 5 共享资产 + 8 分镜 = 13
- 已完成 = 4（4 个共享资产已 ✅）
- 列填 `4/13`

复用让"已完成"数自然跨集累积——第2集开始时就有 4 个起步分。

### 19.6 为什么这种架构？

- **避免重复**：4 张定妆图存 100 次 = 400 张 PNG。共享后 = 4 张。
- **保证一致**：跨集偶然修改某张定妆会立刻让其他集不一致。集中管理避免漂移。
- **降低生成成本**：复用 = 无需重抽，省钱（即梦会员积分）省时间。
- **方便回溯**：所有角色"曾经画过什么样"集中在 `出图/共享/prompt/角色定妆.md`，无需翻 N 个集子。

### 19.7 跟 `characters/` / `locations/` 的关系

- `设定库/characters/角色名.md` 是**人物设定文档**（小说级，剧情服务）
- `出图/共享/prompt/角色定妆.md` 是**生产实战 prompt**（AI 服务，含即梦操作 SOP）
- 前者是设定，后者是执行。前者由编剧维护，后者由制作维护。
- 两者锁定的"妆造拆解"完全一致——`出图/共享/图片/` 只是把 `characters/` 的 ① 定妆照 prompt 加上即梦操作、检查清单、参考图依赖等"实战包装"。

## Q20：下一集出料时怎么自动复用前几集的定妆？SOP 是什么？<a id="q20"></a>

### 20.1 强制 5 步 SOP（每集出图 prompt 生成必走）

```
① 扫描共享库
   读 出图/共享/prompt/00_索引.md
   盘清楚：已有哪些角色（含形态变体）/场景/道具，及状态（✅/⏳/⬜）
   ↓
② 列出本集需求
   读 脚本/第N集/分镜剧本.md + 素材清单.md
   提取本集需要的所有 角色/场景/道具/特殊视觉
   ↓
③ 差集 = 新增
   本集需求 − 共享已有 = 必须新加入共享库的项
   包括："首次出现的全新角色/场景/道具" + "已有角色的新形态变体"
   ↓
④ 追加共享库（仅新增项）
   出图/共享/prompt/00_索引.md 追加 ⬜ 行
   出图/共享/prompt/角色|场景|道具定妆.md 追加实战 prompt 块
   ↓
⑤ 建本集 prompt 文件夹
   出图/第N集/prompt/00_总览.md（含引用共享）
   出图/第N集/prompt/01_分镜出图.md（本集独有分镜）
```

### 20.2 实战例子：第2集 出图 prompt 生成

| 步骤 | 第2集 实际操作 |
|---|---|
| ① 扫描 | 共享库已有：沈念常态 ✅、沈念觉醒态 ✅、小禾 ✅、柳娘子 ✅ |
| ② 列需求 | 本集需要：沈念常态、小禾、偏殿修炼室、系统光幕、12 个分镜场景 |
| ③ 差集 | **新增**：偏殿修炼室 LOC_02、系统光幕（共享道具已占位但 prompt 未写满）|
| ④ 追加 | 共享 `场景定妆.md ②` 占位 → 完整 prompt 块；`道具定妆.md ②` 占位 → 完整 prompt 块 |
| ⑤ 建本集 | `出图/第2集/图片/00_总览.md` 引用共享 + `01_分镜出图.md` 12 镜 |

**复用红利**：第2集开局 **2/16**（沈念常态 + 小禾 直接复用）

### 20.3 跨集累积效应

| 集 | 开局复用 | 新增 shared | 本集分镜 | 出图列 |
|---|---|---|---|---|
| 第1集 | 0 | 4 角色 + 1 场景 + 1 道具 = 6 | 13 | 4/19 |
| 第2集 | 2（沈念常态 + 小禾）| 1 场景 + 1 道具 = 2 | 12 | 2/16 |
| 第3集（预估）| 5+（前面 shared 全继承）| 蛇妖 + 三小妖 + 老槐树 + 枯井 等 | 8-12 | 起步分高 |
| ... |越往后越累 | 越往后越少 | — | 分母小、分子高 |

到第10+ 集，新增 shared 基本只剩"该集独有的小角色"，每集做 prompt 只需 1-2 张 shared + 8-12 分镜。

### 20.4 跳过第 ① 步的代价（铁律）

```
不扫共享 → 把已存在的定妆当"新"建 → 跨集脸漂移 + 浪费即梦积分 + prompt 库重复
```

无论什么场景，**任何集生成出图 prompt 之前必先扫共享**。这条写进 `SKILL.md` 第3步 item 8 的"强制 SOP"。

### 20.5 死亡角色 / 一次性形态 还放共享吗？

**放**。统一规则 > 节省空间。柳娘子第1集就死了也放 shared——一致性比省 3MB 重要，且未来可能闪回。

### 20.6 跨集判定"是否复用"的 3 个常见误判

| 看起来"是同一个"实际可能不能复用 | 怎么处理 |
|---|---|
| 同一角色的不同形态（觉醒态 vs 银牌态）| **算新增**，建独立 shared 条目 |
| 同一场景的不同时间/光线（白昼 vs 夜晚）| 多数情况可以一张 PNG，分镜 prompt 改光线即可。若反差太大（晨光 vs 暴雨）才分两张 |
| 道具的不同状态（完好木梳 vs 碎成粉末木梳）| 看是否反复入镜——一次性的不进 shared，反复的才进 |

## Q23：用 gemini 出图行不行？为什么免费号和免费 API key 都出不了图？<a id="q23"></a>

### 23.1 一句话结论
**图像生成是付费能力**——无论 gemini CLI 免费个人号、还是免费档 API key，都给不了图。判断任何"图 AI 能不能用"，先验**权限 + 配额**，别只看模型"可见"。

### 23.2 gemini CLI（免费 oauth-personal 个人号）
- headless 本身通（`-p` 非交互 + `-y` YOLO 自动批准工具，连得上 API，鉴权走文件 `~/.gemini/oauth_creds.json` 无钥匙串问题）。
- **但所有图像模型 404 `ModelNotFoundError`**（`gemini-2.5-flash-image` / `gemini-3.x-image` / `imagen-4.0`）；默认文本模型自答"我无法直接生成图像文件"；无图像扩展/skill。→ 免费 OAuth 没被授予图像模型。

### 23.3 gemini API key（免费档）
- `curl .../v1beta/models?key=` 能列出 54 个模型，含 `gemini-2.5-flash-image`、`gemini-3.1-flash-image`、`imagen-4.0-*`、`veo-*`——**"可见"≠"可用"**。
- 真打：`generateContent` → **429 `generate_content_free_tier_requests, limit: 0`**（免费档图像配额为 0）；imagen `predict` → **400 "Imagen only available on paid plans, upgrade at ai.dev/projects"**。

### 23.4 解法
- **① 给 key 开 billing**（ai.dev/projects 升付费）后，REST 直调最稳：`POST .../models/gemini-2.5-flash-image:generateContent`，body `generationConfig.responseModalities=["IMAGE"]`（+可选 `imageConfig.aspectRatio`），从 `candidates[].content.parts[].inlineData.data` 取 base64 存 PNG。脚本见 `出图/_gen_gemini.py`（支持参考图走 inlineData、aspectRatio 回退、~$0.039/张 nano banana）。key 存 `~/.gemini/.apikey`、用 `$(cat …)` 读、**不进对话/不打印**。
- **② 图片阶段不再用即梦额度兜底**：即梦 dreamina 额度仅保留给视频阶段；正式出图按 `生图AI` 选择的官方后端执行，项目内统一。
- nano banana 默认偏西方脸 → image prompt 末尾拼"中国古代东方面孔"等锚定句（见 Q18 + 共享 `00_索引.md`）。

## Q24：稳定产线里，定妆照 / 场景图 需不需要"角色训练"（LoRA）？怎么训？<a id="q24"></a>

### 24.1 铁律：要不要训练 = 你最终在哪个平台出图
**即梦 / 可灵 / Gemini 都是闭源，不接受你自训的 LoRA**；自训模型只能在**开源生态（SD / Flux / ComfyUI）**出图。所以这是个绑定平台的决策。

### 24.2 三档一致性方法（由轻到重）
1. **无训练 · 参考图法**（主流，你现在的路线）：定妆照 → 在 `生图AI` 所选官方后端中作为多图参考反复复用（正面主参考 + 侧面 + 背面 + 半身/全身服装参考 + 场景参考）。零训练、最少迁移成本；一致性"够用但非完美"，复杂角度/光照偶漂。
2. **平台内置"建角色/形象"**：上传 N 张图建一个可复用角色 ID，平台内、无需 GPU，比纯参考图稳。**强烈建议用上**。
3. **真 LoRA / DreamBooth**（仅开源生态）：开源底模 + 该角色 15–30 张多角度图训一个 LoRA，云/本地训练，ComfyUI + LoRA + ControlNet/IPAdapter 出图。一致性最强、可控性最高；代价是离开即梦/可灵、要 GPU/云训练。

### 24.3 ROI 拐点
角色**反复出现几十集以上 + 对脸要求已超出参考图能压住** → 才值得训 LoRA。
例（《冷宫有妖气》）：女主**沈念**（贯穿全篇 + 常态/觉醒/银牌多形态）= 最该训；只出 1–2 集即死的**柳娘子 / 三小妖** = 纯参考图，绝不值得训。

### 24.4 场景图
**基本不用训练**——场景对一致性容错远高于人脸，靠"场景定妆图 + 风格参考（强度 0.4–0.5）+ 统一画风词"即可。

### 24.5 推荐【混合产线】分工
- Codex/OpenAI 负责正式出图；视频后端（即梦/可灵/Seedance/Veo）只负责图生视频运动；
- 仅 **1–3 个核心长线角色**在开源（Flux/SDXL）训 LoRA、用 ComfyUI 出他们的关键镜头，再回流统一画风。

### 24.6 经来源核实的结论（2026，deep-research 验证）

**A. ROI 拐点**
训练判据 = 角色长期反复出现 + 质量要求高 + 需跨多场景；"一次性/测试镜头跳过训练，用图生图"。沈念这类全篇核心角色**该训**，柳娘子/三小妖**不训**。

**B. 闭源平台"建角色"真相**
- **即梦 Dreamina（Seedance 2.0）= "全能参考"多模态引用，不是训练一个持久角色**：单次生成可混引最多 **9 张图 + 3 段视频 + 3 段音频**，用 `@素材名` 维持面部/服装/体型；多镜头连贯靠"上段尾帧→下段输入"。即**强化版参考图**，无独立"角色库"。
- **可灵 Kling 两套**：
  - **Custom Model**：上传 **10–30 段视频**（每段≥10s，注意是**视频不是静态图**）训练可复用角色，`@tag` 跨生成复用、一致性高。
  - **Element Library / "Bind Subject"**：约 **4 张参考图**定义一次角色，跨多序列复用。

**C. 开源 LoRA how-to**
- **底模**：用 **Flux**（FLUX.1-dev），工具 **Ostris AI-Toolkit / FluxGym**。
- **数据集**：**15–20 张甜区**（15–50 可用）；按景别均衡，**1024×1024**；已有定妆照可直接当种子。
- **云训练成本**：RunPod RTX 4090(24GB) **Community $0.34/hr、Secure $0.69/hr**；**fal.ai FLUX LoRA fast ≈ $2/次**。显存：Flux LoRA 24GB 最低。
- **出图链**：ComfyUI 里 **角色 LoRA 强度 0.7–1.0** + **IP-Adapter FaceID 控脸（3–5 张好参考）** + **ControlNet 控姿势**。

## Q25：产线顺序大重排 —— 配音前移、时长驱动镜头（2026-06 落地）<a id="q25"></a>

**起因**：第1集成片时发现「配音迁就预设字幕窗口被迫压速、不同步」。深挖根因——**当时产线把配音放在最后，镜头时长按平台默认估**（故事板写死 6-8s、字幕时间轴从估值推），与主流工作室「先配音、配音时长决定镜头时长」**正好相反**。

**修正后的六阶段产线**（已合入 main）：
```
小说 → n2d-script阶段1·剧本改编(voiceover台词 + bgm + 封面，**不做分镜**)
     → n2d-voice(角色配音 + 统计每句台词时长·时长清单.json)
     → n2d-script阶段2·分镜设计(时长驱动 → 分镜剧本 + 故事板Clip时长 + 素材清单 + 字幕SRT + 镜头时长.json)
     → n2d-image(出图) → n2d-video(图生视频·clip落 出视频/第N集/视频/)
     → n2d-compose(剪辑合成 + 背景音乐 + 字幕 → 成片_第N集_{mode}.mp4)
```

**关键新增**：
- **`时长清单.json`**（n2d-voice 产）= 配音驱动镜头的桥梁。`finalize_storyboard.py` 读它 → 重定时 SRT + `镜头时长.json` → 锁故事板 Clip 时长；单 Clip 超即梦 8s 自动标拆。
- **n2d-voice**：多后端可插拔（CosyVoice/GPT-SoVITS 本地克隆 > MiniMax/火山 云 > macOS say 占位），含声音克隆。
- **n2d-compose**：Homebrew ffmpeg **无 libass** → 用 **Pillow 渲染字幕 PNG + overlay 烧录**；配音 + ducking BGM + clip 自带音效底 三层混音。

## Q26：怎么让 n2d 的 prompt 更有"导演视角"（镜头/剪辑节奏/配音/一致性）？<a id="q26"></a>

**起因（2026-06-03）**：观察到**红果爆款的画质很多不是行业顶级，但镜头 + 剧情节奏做得好，观众愿意一直追**。要把这套"留人"的导演视角显性化进 prompt——从四个角度：镜头设计 / 剪辑节奏 / 配音 / 角色一致性。

### 26.1 先诊断：哪些已经强，哪些是真缺口

| 角度 | 现状 | 缺口 |
|---|---|---|
| **镜头设计** | ✅ 强。景别系统 / 180°轴线 / 30° / eyeline / 转场 / 运镜克制 | 基本够用 |
| **角色一致性** | ✅ 强。定妆库 + 角色卡锚点句 + 共享层 + 每镜拼锚点 | 基本够用 |
| **剪辑节奏** | ⚠️ 散且弱。n2d-compose 是**纯拼接**，无卡点、无镜头时长曲线 | **最大缺口** |
| **配音** | ⚠️ 偏工程。情绪标签被丢弃，没有表演（语速/停顿/重音） | 表演维度缺 |

### 26.2 解法：补一条贯穿全程的「导演节奏」主线（1 纲领 + 6 改造）

**新建留存圣经** `n2d/references/导演节奏.md`：`分镜语法` 管**镜头怎么切（空间）**，`导演节奏` 管**节奏怎么走（时间·留存）**。

六处改造：
1. **n2d-script 阶段1**：写词前先设计"留存曲线节拍点表"（开场钩/钩子/爽点/集尾钩各在哪）。
2. **n2d-script 阶段2**：`故事板.md` 每 Clip 标节奏注记（铺垫·长镜 / 加速·碎切 / 爽点·CU硬切 / 留白·定格）。
3. **n2d-voice**：`render_voice.py` 解析每句情绪→TTS；`\|\|`→停顿气口；钩子句后留"悬念呼吸"拍。
4. **n2d-video**：prompt 增 **节奏/张力** 维度，运镜服务情绪（逼近=推 / 高光=环绕 / 释放=拉）。
5. **n2d-compose**：明确**不许把 clip 等长化**（时长曲线就是剪辑节奏）；新增 `BGM_OFFSET` 让 BGM drop 卡在爽点那一帧。
6. **n2d 调度**：挂出 `导演节奏.md`。

## Q27：先配音还是先视频？占位怎么用？后期剪辑删减怎么办？<a id="q27"></a>

### 27.1 先配音，不要先视频（既定铁律）
视频比图贵 1-2 个数量级，镜头多长该由台词多长决定。
- 视频先行 → 拍脑袋估时长 → 真配音对不上 → **重切镜头、重出视频**。
- 配音先行 → 锁死 SRT + `镜头时长.json` + 故事板 Clip 时长 → 再出图/出视频。

### 27.2 "占位"分阶段——别用占位时长喂出图/出视频
**占位时长 ≠ 真实配音时长。** 占位只服务"出图之前"；跨过出图线，时长必须是真的。

### 27.3 后期删减：在源头删 + 向下游回流
删减要**回源源改、重跑回流**，不在成片上剪。
首选：把删减提前到剧本/配音阶段。已出图/视频要删时，用 `delete_shot.py` 自动重定时。

## Q28：「逐镜出图→出视频」vs「故事板/定妆库先行」哪种更好？<a id="q28"></a>

**结论**：**后者明显更好，它是前者的成熟形态**。
- 前者：每个分镜各自 text2image，镜头之间相对独立。
- 后者：先做**人物定妆张 + 场景图**当锚点，再用多图参考把每个镜头图"派生"出来。

**为什么后者更好**：
- **视觉一致性**是漫剧的核心质感，漂移 = 观感破产。
- 后者是目前唯一能压住漂移的工程化做法（跨场景相似度可达 95%）。
- 优先级阶梯：**① 锚点句 > ② 定妆多图参考派生（多视图） > ③ 扫共享库 > ④ 不中途换工具 > ⑤ 平台建角色/LoRA**。

## Q30：从零做一部完整 AI 漫剧——推荐的一步步出脚本/出图/出视频顺序（总纲）<a id="q30"></a>

1. **剧本改编**：拆集 + 台词 + 角色/场景卡（不做分镜）。
2. **配音（前移）**：真实音色配音，拿 **`时长清单.json`**。

5. **出视频**：clip时长=配音驱动，图生视频。
6. **合成**：配音 + BGM + 烧中英字幕。

## Q34：能不能"先出视频、后期再配音"？（快速 demo 模式 vs 配音先行）<a id="q34"></a>

## Q38：小说怎么拆？「集 → 镜头(分镜) → Clip」三层粒度各自的依据是什么？<a id="q38"></a>

3. **Clip**：相邻分镜合并，时长不超后端上限，换场处必断。

## Q39：故事板是怎么来的？就是把几个镜头放一起吗？<a id="q39"></a>

**结论**：分组只是表皮，**故事板真正的身份是契约层**。
它定下了：
- **continuity**：接缝接力（首/尾帧）。
- **visual_contract**：光位/轴线/状态。
- **style_contract**：基调/运动边界。
视频改不动的导演决策都在这层定死。

## Q41：即梦 multiframe2video（智能多帧）怎么用？跟中段锚帧什么关系？<a id="q41"></a>

**结论**：`multiframe2video` 是原生多关键帧能力（20帧出连续镜）。
- **原生多帧**：把 `[首帧, 中锚1..K, 尾帧]` 一次喂入，自动运镜，无焊缝，不翻倍成本。
- **三帧契约**：默认出一张 `_mid` 中锚供多帧引用。
这套干净链路取代了旧的“拼贴板”模式。
