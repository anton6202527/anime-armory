# song-score — 歌词与概念早期体检

在调用 Suno / ACE-Step / DiffRhythm 消耗积分或算力生成音频**之前**，先对 `词/lyrics.md` 和 `创作蓝图.md` 进行打分。

## 为什么要跑 song-score？
生成音频的成本远高于生成文本。如果歌词拗口、缺乏 Hook、或者结构混乱，生成的歌曲大概率不好听。`song-score` 充当质量闸门，确保喂给后端模型的词是"可唱且有潜力"的。

## 评分维度 (1-10分)
1. **商业潜力与 Hook**：副歌（Chorus）是否有极强的记忆点？是否符合当前短视频或流媒体的传播规律？
2. **结构与对称性**：主歌（Verse）之间字数是否对称（方便套用旋律）？桥段（Bridge）是否起到了推波助澜的作用？
3. **押韵与流畅度**：韵脚是否自然？会不会为了押韵而造词？
4. **情感共鸣**：词意是否贴合蓝图设定的情绪基调？能否打动目标受众？

## 工作流
1. **自动化机检**：执行 `python3 skills/song-craft/scripts/analyze_lyrics.py <作品根>`，提取结构对称性和韵脚硬伤（如 Verse 1 和 Verse 2 行数不一致）。
2. **LLM 语义评分**：结合机检结果和 `词/lyrics.md` 的内容，让 LLM 按照上述四个维度打分。
3. **决策与回流**：
   - **高分 (≥80)**：可以放心推进到 `song-compose`。
   - **中分 (60-79)**：局部修改（如微调某句的字数或替换某个不押韵的词），然后推进。
   - **低分 (<60)**：打回 `song-lyrics` 重新梳理 Hook 和结构。

## 执行入口
该过程主要依赖 `analyze_lyrics.py` 提供客观数据，由 LLM 完成主观评价。

```bash
python3 skills/song-craft/scripts/analyze_lyrics.py <作品根>
```
然后让 LLM 阅读输出和歌词文件，给出评分与修改建议。

## 常见错误

| 错误 | 纠正 |
|---|---|
| 不等 LLM 判断，纯靠脚本就打回歌词 | 机检脚本只提取“对称性”和“硬伤数据”，真正的商业潜力和情绪打分需要 LLM 辅助分析 |
| 因为小扣分就全盘废弃 | 60-79 分中段区间应该指导用户微调（换韵脚或改某句长度），而不应轻易废弃整个副歌 Hook |
